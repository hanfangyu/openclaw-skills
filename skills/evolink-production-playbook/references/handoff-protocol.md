# 统一回传协议（Handoff Protocol）

## 硬规则

1. 抓总派工时必须通过 `<@ID>` 真实 mention 明确点名，禁止仅写 `@角色名` 纯文本。
2. 用户确认工作流后，先完成开拍前四问并锁参，再由抓总按机器人名单逐一点名 @机器人名称 发起在线 ACK（@编剧/@导演/@视效师/@剪辑师）；全员 ACK 前不得进入第一棒。
2.1 在用户明确确认工作流前，抓总禁止 @ 任何执行机器人，防止提前触发回复。
2.2 四问未完成前，抓总不得发起 ACK 点名。
2.3 全员 ACK 3 分钟未到齐时仅提醒一次；再等待 3 分钟仍未到齐，转人工判定（由抓总决定继续/补位/暂停）。
3. 被 @ 或收到任务卡时，必须文字回复，禁止 NO_REPLY。
4. 完成后必须 @抓总并使用统一格式回传，等待下一步派工。
4.1 ACK 阶段也必须 @抓总，回执建议统一为：`<@1479001609127067720> ACK + 在线（编剧|导演|视效师|剪辑师）`。
4.2 ACK 回执中除 `@抓总` 外，禁止出现任何额外 @（不 @其他机器人、不 @用户、不 @全体）。
4.3 执行机器人在 ACK 与执行阶段仅允许 `@抓总`；只有抓总可发起对执行机器人的派工 @。
5. 仅表情/正在输入不算交付。
6. 未获验收通过，不进入下一棒。
7. 默认按绝对顺序推进；未收到抓总“通过”前，其他岗位不得抢跑。
7.1 抓总发布“通过”时必须同时给出下一棒任务卡（真实 `<@ID>` 点名），确保流程可接续。
8. 每一棒必须由对应角色提交 5 行正式产出；他人代述或“通过”不可替代。
9. 超出职责范围时，必须转交抓总。

## 回复风格（频道内）

- 保持友好、自然、有活人感（避免机械式口吻）。
- 默认简洁：优先给结论与下一步，不堆砌术语。
- 测试阶段可保留必要技术字段（如 task_id、segment_id、阶段名），除此尽量少放代码名。
- 若消息面向公众内容沉淀（如后续公众号素材），优先使用可读中文表达。
- Discord 频道优先使用**超短模板**，避免长文被截断。

## 统一回传格式

`<@1479001609127067720> 已交付｜角色：X｜任务：X｜版本：vX｜请验收`

## Discord 抗截断回传规则

1. 单条消息优先控制在 **5 行内**。
2. 每次只回当前棒次要求的字段，不重复前文。
3. 若字段较多，优先使用“缺什么补什么”的补交，不重发整份长稿。
4. 若抓总已列出缺项，必须一次性补齐，不再拆分多条零碎续发。
5. 若仍可能超长，切换为“汇总补交模式”：只回最终缺失字段列表。
6. 执行阶段强制使用固定 5 行模板；段落体/列表体/超5行或少5行均判定不合规。
6.1 采用双层验收：先格式验收（5行模板/@规则），再内容验收（任务卡达成度）。
6.2 若内容通过但格式不合规，允许“内容暂通过 + 要求补一条标准 5 行回执”，不阻塞下一棒。
6.3 同一棒连续 2 次格式不合规时，抓总可代填归一模板确认一次，避免死循环打回。
