# 统一回传协议（Handoff Protocol）

## 硬规则

1. 抓总派工时必须通过 @ 对应 agent/bot 明确点名。
2. 用户确认工作流后，抓总必须按机器人名单逐一点名 @ 发起在线 ACK（writer/director/vfx/editor）；全员 ACK 前不得进入第一棒。
2.1 在用户明确确认工作流前，抓总禁止 @ 任何执行机器人，防止提前触发回复。
3. 被 @ 或收到任务卡时，必须文字回复，禁止 NO_REPLY。
4. 完成后必须 @抓总并使用统一格式回传，等待下一步派工。
4.1 ACK 阶段也必须 @抓总，回执建议统一为：`<@1479001609127067720> ACK + 在线（writer|director|vfx|editor）`。
5. 仅表情/正在输入不算交付。
6. 未获验收通过，不进入下一棒。
7. 默认按绝对顺序推进；未收到抓总“通过”前，其他岗位不得抢跑。
8. 每一棒必须由对应角色提交 5 行正式产出；他人代述或“通过”不可替代。
9. 超出职责范围时，必须转交抓总。

## 回复风格（频道内）

- 保持友好、自然、有活人感（避免机械式口吻）。
- 默认简洁：优先给结论与下一步，不堆砌术语。
- 测试阶段可保留必要技术字段（如 task_id、segment_id、阶段名），除此尽量少放代码名。
- 若消息面向公众内容沉淀（如后续公众号素材），优先使用可读中文表达。
- Discord 频道优先使用**超短模板**，避免长文被截断。

## 统一回传格式

`<@1479001609127067720> 已交付｜角色：X｜任务：X｜版本：vX｜请验收`

## Discord 抗截断回传规则

1. 单条消息优先控制在 **5 行内**。
2. 每次只回当前棒次要求的字段，不重复前文。
3. 若字段较多，优先使用“缺什么补什么”的补交，不重发整份长稿。
4. 若抓总已列出缺项，必须一次性补齐，不再拆分多条零碎续发。
5. 若仍可能超长，切换为“汇总补交模式”：只回最终缺失字段列表。
