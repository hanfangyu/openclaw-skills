# openclaw-team-chronicle

把当天真实发生的事，写成一篇可以公开发到公众号的团队连载纪实。

默认叙事起点不是“团队诞生”，而是“团队已经在场，但正在被重新培训、重新磨合、重新定形”。

## Workflow

1. 先找当天素材，不先定题。
2. 从素材里提炼出当天最有张力的一件事。
3. 决定由谁来讲今天这件事。
4. 写成故事，而不是教程。
5. 如果用户要落成可发布稿，再补齐公众号需要的 frontmatter。

## Source Scan

优先读取这些文件，按顺序尽量少读：

1. `memory/YYYY-MM-DD.md`
2. `memory/<昨天>.md`
3. `AGENTS.md`
4. `ROLE.md`
5. `USER.md`

只在需要角色气质或长期背景时，再补读：

- `MEMORY.md`
- `IDENTITY.md`
- `SOUL.md`
- 角色设定类文件，例如 `CAST.md`、`PERSONA.md`、`team/*.md`

默认只提炼这 5 类信息：

- 今天抓总给了什么目标
- 今天最卡的一件事是什么
- 谁站出来接住了问题
- 谁暴露了短板或性格
- 明天最值得继续看的悬念是什么

不要为了凑文章去扫一堆无关文件。

## Narrative Rules

文章必须满足这些约束：

- 写成“团队成长纪实”，不要写成教学贴
- 真实取材，可以压缩和重组，但不要编造实际没发生的结果
- 少讲技术名词，多讲人、气氛、决断、失误、补位、关系变化
- 每篇都要有一个清晰的当天主冲突
- 每篇结尾都要给出明天还能接着看的钩子

默认视角：

- 首选 `抓总` 第一人称
- 如果当天某个成员最有戏，也可以切成该成员视角或近距离旁白
- 不要一天换很多讲述者，保持单篇视角稳定

默认叙事语气：

- 像办公室群像，不像项目汇报
- 像连续剧纪实，不像教程
- 克制、具体、有画面，不要鸡血口号

## Language Rules

优先用故事词替换技术词。可参考 [references/language-map.md](references/language-map.md)。

硬规则：

- 不把文章写成“我们今天完成了 1、2、3”
- 不堆叠“配置、编排、调用、链路、参数”这类词
- 不输出大段流程说明
- 不把成员写成同一种口气

## Story Spine

默认用四段骨架：

1. 开头先给现场感或冲突
2. 中段写今天真正发生的推进
3. 后段写人物变化或团队关系变化
4. 结尾留一个明天的钩子

如果当天素材很弱，也要保留这四段，只是缩短篇幅。

## Daily Output

默认输出以下内容：

1. `title`：当天动态长出来的标题，不提前预设选题库
2. `summary`：一句话摘要，可用于公众号导语或卡片文案
3. `article`：正文 Markdown
4. `tomorrow_hook`：一句话说明明天最值得继续写什么

正文默认长度：

- 800 到 1500 字中文为宜
- 当天事件很集中时可以更短
- 不为了凑字数硬加解释

标题生成规则：

- 从当天最强冲突、最明显变化、最像“转折点”的事件里长出来
- 不要标题党
- 不要像教程题目
- 可以带一点人物感、办公室戏感、轻连载感

## Daily Run Defaults

如果用户没有指定日期，默认取本地今天。

如果当天 `memory/YYYY-MM-DD.md` 还不存在：

1. 读取最近一篇日记
2. 明确这是“基于当前最新记录写出的今日篇”
3. 不要伪造今天没发生的推进

如果用户没有指定输出路径：

- 默认先在回复里直接给成稿
- 只有用户明确要求落盘时，才写入 Markdown 文件

## WeChat Draft Mode

如果用户明确要“保存成公众号稿”或“直接交给微信发布技能”，则输出 Markdown 文件并补齐 frontmatter：

```markdown
---
title: 动态生成的标题
cover: <用户提供的封面图或已确认的默认封面>
---
```

注意：

- `cover` 不能乱写
- 如果用户没有给封面，也没有稳定默认封面，就先产正文并说明“还差封面”
- 只有当用户明确要求发布时，才继续衔接 `wechat-publisher`

## Guardrails

- 不要提前为未来几天拟整套题目
- 不要把没发生的对话当成真实对话直写出来
- 如需写人物对白，只能基于当天真实表达做压缩改写，并保持可信
- 不要把一次普通修改夸张成史诗剧情
- 不要把文章写成夸夸文，必须保留当天的问题和不顺
- 如果当天没什么推进，就写“停滞、误解、等待、修正”本身

## When The User Gives A Stage Goal

如果用户只给了一个阶段目标，例如：

- 重新培训团队
- 给成员定角色形象
- 准备一次公开亮相
- 团队第一次真正协同

则把这个目标当成“本周主线”，但当日文章仍然只写当天真的发生了什么。

不要把“阶段目标”直接翻成生硬计划书。

## Hand-Off

交付时优先给出成稿，不要先长篇解释你的写法。

如果用户要继续发公众号，补充两样短信息即可：

- 一句封面图建议
- 一句次日连载钩子
