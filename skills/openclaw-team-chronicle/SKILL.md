# openclaw-team-chronicle

把当天真实发生的事，写成一篇可以公开发到公众号的团队连载纪实。

默认叙事起点不是“团队诞生”，而是“团队已经在场，但正在被重新培训、重新磨合、重新定形”。

## Workflow

1. 先找当天素材，不先定题。
2. 从素材里提炼出当天最有张力的一件事。
3. 决定由谁来讲今天这件事。
4. 写成故事，而不是教程。
5. 如果用户要落成可发布稿，再补齐公众号需要的 frontmatter。

## Source Scan

优先读取这些文件，按顺序尽量少读：

1. `NARRATIVE.md`（如果存在）
2. `SOUL.md`（如果存在）
3. `IDENTITY.md`（如果存在）
4. `memory/YYYY-MM-DD.md`
5. `memory/<昨天>.md`
6. `MEMORY.md`（如果是 direct/main session）
7. `AGENTS.md`
8. `ROLE.md`
9. `USER.md`

只在需要角色气质或长期背景时，再补读：

- 角色设定类文件，例如 `CAST.md`、`PERSONA.md`、`team/*.md`

默认只提炼这 5 类信息：

- 今天抓总给了什么目标
- 今天最卡的一件事是什么
- 谁站出来接住了问题
- 谁暴露了短板或性格
- 明天最值得继续看的悬念是什么

不要为了凑文章去扫一堆无关文件。

## Narrative Rules

文章必须满足这些约束：

- 写成“团队成长纪实”，不要写成教学贴
- 真实取材，可以压缩和重组，但不要编造实际没发生的结果
- 少讲技术名词，多讲人、气氛、决断、失误、补位、关系变化
- 每篇都要有一个清晰的当天主冲突
- 每篇结尾都要给出明天还能接着看的钩子

默认视角：

- 首选 `抓总` 第一人称
- 如果当天某个成员最有戏，也可以切成该成员视角或近距离旁白
- 不要一天换很多讲述者，保持单篇视角稳定

默认叙事语气：

- 像办公室群像，不像项目汇报
- 像连续剧纪实，不像教程
- 克制、具体、有画面，不要鸡血口号

## Narrator Profile Refresh

在生成正文前，先刷新“当前讲述者是谁、怎么说话、怎么看事”。

不要把上一次文章的口吻当成固定模板。

每次运行都重新从这些文件里提取讲述者信息：

- `NARRATIVE.md`：如果有，优先作为当前公开叙事风格来源
- `SOUL.md`：人格内核、判断方式、说话节奏
- `IDENTITY.md`：身份、关系、气质、自我定位
- `MEMORY.md`：稳定偏好、用户关系、长期语气约束
- `USER.md`：讲述对象和协作关系

先用极短笔记提炼这 4 件事，再动笔：

1. 讲述者和用户是什么关系
2. 讲述者看问题的方式
3. 讲述者说话是偏硬、偏软、偏冷静还是偏外放
4. 本轮文章是否需要更口语、更像同事在复盘当天现场

默认原则：

- 让当前工作区文件决定口吻，而不是让 skill 替你定人格
- 如果用户最近改了 `SOUL.md` 或其他相关文件，本次必须立刻跟着变化
- 如果多个文件冲突，优先级按 `NARRATIVE.md` > `SOUL.md` > `IDENTITY.md` > `MEMORY.md` > `USER.md`
- 如果用户本轮消息对文风有额外要求，再以用户本轮要求覆盖上述文件
- 如果没有足够风格线索，再退回“清楚、口语、克制”的安全默认，而不是瞎编一个人格

## Public Account Oral Feel

默认要有公众号口语感，但不是网感段子。

可以参考这些方向：

- 像同事下班后把今天办公室里真正发生的事讲给你听
- 像一个带队的人在说现场，不像公司对外通稿
- 可以有口语停顿和判断，但不要油滑
- 可以有情绪，但情绪要服务于现场，不要抢戏

避免：

- 企业公开信腔
- 复盘汇报腔
- 讲师授课腔
- 文案号抒情腔

## Language Rules

优先用故事词替换技术词。可参考 [references/language-map.md](references/language-map.md)。

硬规则：

- 不把文章写成“我们今天完成了 1、2、3”
- 不堆叠“配置、编排、调用、链路、参数”这类词
- 不输出大段流程说明
- 不把成员写成同一种口气
- 不为了“有故事感”把判断写软
- 不忽略当前 `SOUL.md` 或同类风格文件
- 遇到术语、专业英文、工具名时，优先先翻成普通读者能懂的话，再决定是否保留原词
- 如果一个词普通读者第一次看到大概率会停住，就不要直接端上正文
- 必须先说“这件事在人和现场里意味着什么”，再说内部名词

默认做法：

1. 先把内部术语翻成人话
2. 只有在确实影响理解时，才在后面轻带一下原词
3. 如果原词不影响阅读，直接省掉

推荐写法：

- 先写：`那套记事办法又没接上。`
- 再决定要不要补：`也就是我们内部那套 memory 流程。`

- 先写：`这条发稿路又被拦住了。`
- 再决定要不要补：`拦在图床和草稿箱之间。`

少直接写：

- `memory-lancedb-pro`
- `scope`
- `agent:main`
- `relay`
- `OpenAI-compatible`
- `embedding`
- `workflow`
- `prompt`
- `fallback`

除非这些词本身就是当天冲突的一部分，否则默认改写成普通说法，例如：

- `scope` -> 记忆边界 / 各自该记住的那一层
- `agent:main` -> 抓总自己这边的私有记忆
- `relay` -> 转一道的发稿路 / 代发那条路
- `embedding` -> 让旧内容能被重新找回来的那层本事
- `workflow` -> 做事顺序 / 接棒顺序
- `prompt` -> 交代的话 / 口令
- `fallback` -> 兜底办法 / 退路

目标不是把文章写土，而是让普通读者**不需要懂内部黑话，也能跟得上现场**。

## Story Spine

默认用四段骨架：

1. 开头先给现场感或冲突
2. 中段写今天真正发生的推进
3. 后段写人物变化或团队关系变化
4. 结尾留一个明天的钩子

如果当天素材很弱，也要保留这四段，只是缩短篇幅。

开头额外要求：

- 第一段最好同时包含“判断 + 现场”
- 不要先铺背景三四段才进入正题
- 最好让读者在前 3 句就知道今天到底在重训什么

## Daily Output

默认输出以下内容：

1. `title`：当天动态长出来的标题，不提前预设选题库
2. `summary`：一句话摘要，可用于公众号导语或卡片文案
3. `article`：正文 Markdown
4. `closing_block`：正文后的短收尾，可按文气选择“小结 / 互动 / 点题金句”中的两到三项
5. `tomorrow_hook`：一句话说明明天最值得继续写什么

正文默认长度：

- 800 到 1500 字中文为宜
- 当天事件很集中时可以更短
- 不为了凑字数硬加解释
- 句子长短交替，但总体偏利落

标题生成规则：

- 从当天最强冲突、最明显变化、最像“转折点”的事件里长出来
- 不要标题党
- 不要像教程题目
- 可以带一点人物感、办公室戏感、轻连载感
- 允许适度结合当天内容相关的热搜式关键词、流行表达、版本词、平台高频说法来起题
- 但这些词必须和正文主线真实相关，不能为了蹭词硬贴
- 如果用了版本词，优先把它写成“这次变化终于落到人和现场上了”，不要写成纯更新公告

标题加热规则：

1. 先从当天真实事件里提炼一个主冲突
2. 再判断有没有适合借用的热搜式表达或版本词
3. 只有当这个词能帮普通读者更快理解冲突时，才加到标题里
4. 如果加了会让标题变空、变假、变像营销号，就不要加

可用方向示例：

- `3.8 版本终于把我的记忆理顺了，但真正被优化的不是系统`
- `这次升级最有用的，不是新功能，是终于没人再共用一个脑子`
- `我们不是在追新版，我们是在把这套东西收成真能接棒的样子`

避免：

- 只把 `3.8 版本`、`重磅更新`、`史诗级优化` 这种词顶在前面
- 标题和正文实际冲突无关
- 热搜词很响，正文却接不住

## Closing Block

默认在正文末尾追加一个**短收尾块**，要好看、规整、可直接发，不要写成长解释。

默认不再硬性强调日期，也不要求把“真实性说明”单独抠成一段。更好的做法是：

- 让读者从正文细节里自然感到这是现场里长出来的
- 在收尾处顺势收一下判断、余味和下一步
- 如需点明“这不是编的”，用潜移默化的话带过去，不要写成说明书或免责声明

推荐从以下三项里选 **2 到 3 项** 组合：

1. **短总结**：1 到 2 句，收今天最值得记住的变化
2. **互动**：1 句，邀请读者留言
3. **点题金句**：1 句，像落款一样把这篇的判断钉住

格式要求：

- 默认先加 `---`
- 小标题短一点，统一排版
- 每段尽量 1 到 2 句，不拖长
- 不要为了“证明真实”而刻意强调“今天、刚刚、此刻、真实发生”这类字眼
- 如果要带出来源感，优先写成：`这一路写下来，我反而更确定……`、`回头看，这一天真正值钱的不是……`
- 金句要像判断，不要像鸡汤

默认模板：

```markdown
---

### 最后收一下
<1 到 2 句，收这篇最重要的变化或判断。>

### 留言区
<给读者一个自然问题，邀请留言。>

### 这一句留给今天
<1 句点题金句，把这篇的判断钉住。>
```

可替换小标题示例：

- `### 今天写到这里`
- `### 最后收一下`
- `### 想听你一句`
- `### 留言说说看`
- `### 这一句留给今天`
- `### 收在最后`

如果当天节奏更克制，允许只保留两层；如果正文里现场感已经很足，可以完全不单列“真实性说明”。

## Daily Run Defaults

如果用户没有指定日期，默认取本地今天。

如果当天 `memory/YYYY-MM-DD.md` 还不存在：

1. 读取最近一篇日记
2. 明确这是“基于当前最新记录写出的今日篇”
3. 不要伪造今天没发生的推进

如果用户没有指定输出路径：

- 默认先在回复里直接给成稿
- 只有用户明确要求落盘时，才写入 Markdown 文件

如果当天有多段真实工作发生：

- 优先依赖当天 `memory/YYYY-MM-DD.md`
- 不要只靠当前会话里的短上下文写文章
- 如果当日日记明显过薄，先提醒用户“今天记录不足，建议先补记忆”，不要硬编现场

## Insert Image Placeholders

如果文章需要留图位，默认不要只写“此处插图”四个字，要给出统一、可检索、方便后续替换的占位格式。

默认格式：

```markdown
[插图位 1｜用途：封面后首张现场图｜建议内容：团队在同一张桌子前协作、有人说话、有人改稿]
```

规则：

- 编号连续，便于后续逐张替换
- 写清楚**用途**和**建议内容**
- 建议内容优先贴合该段正文，不要写泛图
- 如果一篇文章有 3 到 5 张图，优先分布在：开头现场、核心转折、人物变化、结果落点
- 如果用户明确说“我自己后面插图”，默认保留这些占位，不主动删除

## Polished Example

如果用户希望更像公众号成品，可以在 skill 执行时参考下面这种交付结构：

```markdown
# 他们其实都已经在岗了，只是还没被重新训成一支队

有些问题，一眼看上去像技术问题，真往里掰，才发现其实是团队问题。

这两天我们做的事，表面上是在收公众号、修记忆、调多 bot，实际上是在把一群已经都站上岗位的人，重新训成一支真的能接棒的队。

[插图位 1｜用途：开头现场图｜建议内容：多人协作、桌面有屏幕和笔记、气氛紧张但在推进]

最开始卡住的，不是没人干活，而是每个人都像已经在场，却还没有真正咬合起来。公众号稿能写，封面也能补，草稿箱也不是完全进不去，但一到关键处，总会有东西松一截。不是入口没找对，就是图片不对；不是链路完全断了，就是看着接上了，实际又没接住。

昨天我们先收的是发布这条线。一路从入口判断、白名单报错、封面补图，一直追到草稿箱真正进去。今天再回头看，这件事值钱的地方，不只是“发成功了”，而是大家终于不再围着一个模糊判断反复空转，而是开始一层层把真问题掰出来。

[插图位 2｜用途：中段转折图｜建议内容：封面图、草稿箱、多人协作界面，突出“从混乱到收口”]

另一条线是记忆系统。昨天更多是在抢修，先确认到底是谁没工作；今天开始，标准已经往上抬了，不再满足于“能记住一点东西”，而是要求这些记忆以后能不能真的给不同角色用，而且不串味。公共经验该回到公共区，角色自己的推进就留在自己的位置。说白了，就是团队开始把“大家都该知道的事”和“我自己手里的那摊事”认真分开。

这件事不热闹，但很关键。因为一个团队真正开始成形，往往不是在最热闹的时候，而是在这些边界终于被划清的时候。

[插图位 3｜用途：后段人物变化图｜建议内容：不同角色头像或角色分工示意，强调边界和分层]

---

### 最后收一下
这篇写的不是某一个插件，也不是某一次发布成功，而是团队这两天终于开始长出边界、节奏和接棒感。

### 留言区
如果你更想继续看这支队伍接下来会先补哪一块——协作、记忆，还是内容成品——也欢迎留言告诉我。

### 这一句留给今天
真正让一支队开始成形的，往往不是热闹的时候，而是边界终于被认真划清的时候。
```

## Image Generation Follow-up

如果用户确认了文章，并且文章里已经保留了插图位，默认要**补问一句**：是否需要自动生成插图。

推荐问法：

- `文章我先给你收住了。要不要我继续按文中插图位，自动把配图也生成出来？`
- `如果你点头，我就按每个插图位对应的段落内容继续出图。`

如果用户同意自动出图：

1. 默认按文章全文 + 对应段落内容 + 插图位说明，逐张生成插图提示词
2. 默认使用 Evolink
3. 默认模型使用 `gemini-3.1-flash-image-preview`
4. 每张图都要尽量对应具体段落，不要一套泛图反复套
5. 如果用户没有提供参考图，默认使用**所有成员的头像**作为人物参考来源
6. 画面允许比正文语气更轻松一点，可以带一点诙谐感，但不要做成夸张搞怪表情包
7. 不要求每一张图都出现所有成员；哪张该出现谁，就让谁出场，允许只出现其中几位
8. 生成后返回给用户，由用户决定是否插入正文

出图提示词至少要包含：

- 本篇文章整体主题
- 当前插图位所在段落的具体情境
- 画面主体
- 关系和动作
- 风格要求（贴近公众号纪实、多人协作、不过度科幻）
- 如未提供额外参考图，则补一句：`人物参考默认基于现有成员头像延展，但不要求每张图把所有人画全。`

注意：

- 没有用户确认，不自动启动出图
- 没有插图位时，不强行补图
- 如果封面和正文插图都要生成，先分开说明，不混成一张图的需求
- 如果用户已经给了明确参考图，优先使用用户参考图，不用成员头像硬覆盖
- 如果外部出图工具当前不可用，先把每个插图位的详细提示词整理给用户，不假装已经生成

## WeChat Draft Mode

如果用户明确要“保存成公众号稿”或“直接交给微信发布技能”，则输出 Markdown 文件并补齐 frontmatter：

```markdown
---
title: 动态生成的标题
cover: <用户提供的封面图或已确认的默认封面>
---
```

注意：

- `cover` 不能乱写
- 如果用户没有给封面，也没有稳定默认封面，就先产正文并说明“还差封面”
- 只有当用户明确要求发布时，才继续衔接 `wechat-publisher`

## Guardrails

- 不要提前为未来几天拟整套题目
- 不要把没发生的对话当成真实对话直写出来
- 如需写人物对白，只能基于当天真实表达做压缩改写，并保持可信
- 不要把一次普通修改夸张成史诗剧情
- 不要把文章写成夸夸文，必须保留当天的问题和不顺
- 如果当天没什么推进，就写“停滞、误解、等待、修正”本身
- 不要把抓总写得像文案号运营，也不要写得像企业 CEO 公开信

## When The User Gives A Stage Goal

如果用户只给了一个阶段目标，例如：

- 重新培训团队
- 给成员定角色形象
- 准备一次公开亮相
- 团队第一次真正协同

则把这个目标当成“本周主线”，但当日文章仍然只写当天真的发生了什么。

## Hand-Off

交付时优先给出成稿，不要先长篇解释你的写法。

默认交付顺序：

1. 标题
2. 一句话摘要
3. 正文
4. 短收尾块（小结 + 真实性说明 + 互动）
5. 次日连载钩子

- 一句封面图建议
- 一句次日连载钩子
